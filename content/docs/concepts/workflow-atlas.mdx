# Workflow Atlas

The Atlas surface at `/atlas` is the agentic workflow catalog for your engagement. It organizes every automated workflow by the persona that uses it and lets you inspect, understand, and prototype each step inline.

## What it is

Atlas is not a list of automations — it's a map of intent. Each workflow is a chain of named steps, each step has a type, and each type tells you where human judgment is required and where the agent runs autonomously.

Persona tabs at the top of Atlas filter the workflow catalog to show only the workflows relevant to that role. These personas come from your `systemix.config.yaml`.

## Step types

Every workflow in Atlas is composed of steps with one of five types:

**Input** — a data entry or trigger point. A form the user fills, a webhook that fires, a scheduled cron that kicks off the chain.

**Agent reasoning** — Hermes (or another agent) processes the available context and produces a structured output. No human required; Hermes decides autonomously within its configured autonomy level.

**Router** — a conditional branch. Based on evidence or agent output, the chain continues down one of two or more paths. Routers are deterministic — they evaluate rules, not judgment.

**HITL** — a Human-in-the-Loop gate. The workflow pauses and pushes a card to the queue at `/graph`. A human reviews the agent's recommendation and either promotes it, runs it longer, or kills it.

**Output** — the terminal step. A component is written, a token is updated, a Figma variable is synced, a hypothesis is closed. Outputs are the moments Systemix changes something.

## Inline prototype viewer

Clicking any step in Atlas opens the prototype for that step in a detail pane — right inside Atlas, no navigation required. The prototype shows the actual UI state the user would encounter at that step: the form, the HITL card, the confirmation screen.

The prototype viewer is not a separate surface. Atlas and the prototype viewer are one integrated surface. Atlas is the navigation layer; the prototype viewer is the detail pane.

## Workflow contracts

Each workflow in Atlas is backed by a contract at `contract/workflows/{slug}.mdx`. The contract captures:

- The workflow's intent and the persona it serves
- Each step's type, description, and responsible agent or human
- The current status: `active`, `paused`, or `archived`
- Any open HITL cards referencing this workflow

Hermes updates workflow contracts as runs complete. The Atlas surface reads directly from these contracts — what you see in Atlas is what's actually running.

## Example workflow

```yaml
---
title: "DS token drift → resolve"
persona: engineer
status: active
steps:
  - type: input
    label: "Drift detected"
    trigger: hermes-poll
  - type: agent-reasoning
    label: "Classify drift"
    agent: hermes
  - type: hitl
    label: "Review decision"
    queue: design-system
  - type: output
    label: "Write resolution"
    writes: contract/tokens/{slug}.mdx
---
```

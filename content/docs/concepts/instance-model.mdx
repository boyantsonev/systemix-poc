# Instance model

One Systemix instance per engagement — not per repo, not per app. The entire value of Systemix is a single unified context loop where marketing, business, design, and engineering read from the same evidence base.

## One instance, four surfaces

A Systemix instance exposes four surfaces inside your project:

**Config** — `/config`

View and edit `systemix.config.yaml`. Defines which surfaces are active, which signals are wired in (PostHog, Figma, Vercel, Storybook), and how autonomous Hermes is allowed to be.

**Graph** — `/graph`

Force-graph of the instance topology. Shows a live runtime feed and routes HITL cards to the right role. The graph is your control tower — the one view that tells you what Hermes is doing right now.

**System** — `/system`

Living contract library. Token contracts, component parity reports, hypothesis results — all Hermes-maintained, human-verified. The source of truth for what your design system currently is, not what you intended it to be.

**Atlas** — `/atlas`

Agentic workflow catalog organized by persona. Each persona tab shows the workflows available to that role. Clicking a step opens its prototype inline. Atlas is the navigation layer on top of prototypes; the prototype viewer is the detail pane, not a separate surface.

## Config topology

```yaml
# systemix.config.yaml — root of a monorepo engagement
surfaces:
  - design-system
  - landing
  - gtm
  - product

signals:
  - posthog-eu
  - figma
  - vercel
  - storybook

autonomy: balanced   # conservative | balanced | aggressive
```

`surfaces` determines which Hermes loops are active and what the local app renders. A landing A/B test and a DS token drift both write to `contract/` and surface in the same HITL queue — under different sections, but one queue.

## Contract directory

```
engagement-root/
├── systemix.config.yaml
├── contract/
│   ├── tokens/*.mdx          ← token contracts (drift, clean, missing-in-figma)
│   ├── components/*.mdx      ← component parity contracts
│   ├── hypotheses/*.mdx      ← A/B experiments for any surface
│   └── workflows/*.mdx       ← Atlas workflow contracts
└── .systemix/
    ├── queue.json            ← HITL queue, one per instance
    └── sync-log.json         ← last sync timestamp per signal
```

All surfaces share one `contract/` directory and one HITL queue. That's the point — a DS token drift and a landing page experiment land in the same queue so whoever is on HITL duty sees the full picture.

## ADR-009 superseded

Early design (ADR-009) proposed separate Systemix instances per consuming app. That was wrong. Splitting instances fragments the evidence base — a DS token change that affects both the landing page and the product UI would only register in whichever instance happened to catch it first.

One instance. One context loop. One queue.
